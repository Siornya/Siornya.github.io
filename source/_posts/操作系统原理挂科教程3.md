## 进程的基本概念

### 程序的顺序及其特征

在单道程序环境下，一个程序总是一条指令接一条指令地往下执行，前一个操作做完，后一个操作才开始——这就是**程序的顺序执行**。这种顺序关系可以用DAG来描述（结点表示要做的操作，箭头表示"必须先做完前者才能做后者"）

顺序执行有三个特征：

- **顺序性**：处理机严格按程序规定的顺序执行，每一步的开始都以前一步的结束为前提。
- **封闭性**：程序运行时独占全机资源，资源状态只由这个程序自己改变，不受外界影响；程序一旦开始，结果就不会被其他程序干扰。
- **可再现性**：只要初始条件相同，无论何时运行、运行多少次、是否中途停顿，得到的结果始终一样。

正是这种"独占、有序、结果可复现"的特性，使得单道顺序执行虽然好理解、好调试，但资源利用率很低——CPU 和 I/O 设备只能轮流空等。

为提高利用率而引入**并发执行**，又恰恰打破了封闭性和可再现性，于是需要**进程**来驾驭并发。

### 进程的定义和描述

较典型的进程定义有：

- 进程是程序的一次执行。
- 进程是一个程序及其数据在处理机上顺序执行时所发生的活动
- 进程是程序在一个数据集合上运行的过程，它是系统进行资源分配和并发调度的一个独立单位

**进程与程序的关系**

| 对比项 | 程序 | 进程 |
|---|---|---|
| 状态 | 静态 | 动态 |
| 存在 | 长期存在（存于磁盘） | 暂时存在（运行时才有） |
| 组成 | 仅指令和数据 | 程序段 + 数据段 + PCB |
| 对应关系 | —— | 一个程序可对应多个进程 |

同一个程序在不同数据集上运行，或被多次运行，会对应**多个不同的进程**。

### 进程的特性

进程具有以下特性：

- **结构性**：每个进程都由程序段、数据段和 PCB 三部分组成。
- **动态性**：进程是程序的一次执行过程，有创建、运行到消亡的生命周期。**动态性是进程最基本的特征。**
- **并发性**：多个进程可以同时存在于内存中，并发执行。
- **独立性**：进程是能独立运行、独立获得资源、独立接受调度的基本单位。
- **异步性**：各进程按各自独立的、不可预知的速度向前推进。

### 进程的组成

一个进程实体（进程映像）由三部分组成：

- **程序段**：进程要执行的代码。
- **数据段**：程序运行时用到的数据、变量、工作区。
- **进程控制块 PCB**：进程存在的**唯一标志**。

**进程控制块 PCB**

**PCB 是进程存在的唯一标志**，操作系统利用 PCB 来：

- 描述进程的基本情况（例如 PID、状态、优先级等）
- 描述活动过程（例如运行到哪、等待什么资源）
- 实现控制与管理（调度、切换、撤销等）

## 进程的状态转化

### 三种基本状态

- **就绪态**：进程已获得除 CPU 外的所有资源，只要分到 CPU 就能立刻运行。
- **执行态**：进程正在 CPU 上运行。单核系统中任一时刻只有一个进程处于执行态。
- **阻塞态**：进程因等待某事件（如等待 I/O、等待资源）而暂停，即使把 CPU 给它也没法运行。

### 状态转换

- **就绪 → 执行（进程调度）**：cpu此时空闲，os的短程调度器（cpu调度）从就绪队列挑一个上cpu
- **执行 → 就绪（时间片完/抢占）**：时间片用完了，或者来了更高优先级的进程把它挤了下去
- **执行 → 阻塞（I/O 请求）**：发生了某个事件，这个事件使得该进程的执行受阻（例如进程请求访问某临界资源，而该资源正在被其他进程访问），此时该进程无法继续执行
- **阻塞 → 就绪（I/O 完成）**：资源得到满足，或者某个事件已经发生完毕（注意这里阻塞回到就绪而不是执行）

要特别注意几个**不存在**的转换：

- **就绪 → 阻塞**：不存在。还没运行，谈不上等待事件。
- **阻塞 → 执行**：不存在。事件发生后只能先回到就绪态，重新等调度，不能直接上 CPU。

### 五状态模型

在三态基础上加入两个状态分别是**创建态**和**终止态**：

- **创建态**：进程正在被创建，PCB 已建立但还没进内存就绪队列。
- **终止态**：进程运行结束或异常，等待操作系统善后、回收资源。

五态模型中，引起的状态转换有:

1. NULL-创建态
2. 创建态-就绪态:资源允许的情况下
3. 执行态-终止态:自然结束或者被强行终止
4. 终止态-NULL
5. 阻塞态-终止态:强行终止

### 挂起状态（七状态模型）

当内存紧张或需要调试时，操作系统可以把进程从内存换到外存，这叫**挂起**。挂起的进程不在内存中，不能马上运行。于是又分出两种挂起态：

- **就绪挂起**：在外存，本来就绪，但被换出。
- **阻塞挂起**：在外存，且还在等待事件。

*注意区分"阻塞"和"挂起"：阻塞强调"在等事件"，挂起强调"被换出内存"。*

## PCB

进程控制块是操作系统为每个进程开辟的一块数据结构，记录了管理这个进程所需的全部信息。**系统是"靠 PCB 认识进程的"**：创建进程的实质就是创建 PCB，撤销进程的实质就是回收 PCB；进程之间能被组织成各种队列，靠的也是 PCB。

PCB的作用包括：

- 作为独立运行单位的标志：系统为程序配置PCB后，它才成为合法进程
- 支持进程的间断运行：进程被中断时，CPU 现场保存到 PCB；再次调度时从 PCB 恢复现场继续执行。
- 提供进程管理信息：如程序入口、I/O 相关信息、资源清单等。
- 提供进程调度信息：如状态、优先级、等待时间、已运行时间等。
- 实现进程同步与通信：如信号量、消息队列指针等

### PCB 中的信息

PCB 里的信息大致可以分为四类：

1. **进程标识符信息**
   - **内部标识符**：系统给每个进程分配的唯一编号（PID），方便系统使用。
   - **外部标识符**：由创建者提供的、便于人识别的名字，还包括父进程、用户标识等。

2. **处理机状态信息**：主要是进程切换时要保存的**现场**，包括通用寄存器、指令计数器 PC、程序状态字 PSW、用户栈指针等。进程被切下 CPU 时把这些存进 PCB，重新上 CPU 时再从 PCB 恢复，从而"接着上次的地方继续跑"。

3. **进程调度信息**：调度时要参考的信息，如进程当前**状态**、**优先级**、等待原因、已等待时间、已占用 CPU 时间等。

4. **进程控制和资源管理信息**：程序和数据的内存地址、占用的 I/O 设备、打开的文件列表、以及用于把 PCB 链入队列的**链接指针**等。

### PCB 的组织方式

系统中同时存在很多进程，也就有很多 PCB，需要把它们有序地组织起来。常见方式有两种：

- **链接方式**：把**同一状态**的 PCB 用链接指针串成一个队列。例如所有就绪进程组成**就绪队列**，等待同一事件的进程组成各自的**阻塞队列**。状态转换时，本质就是把 PCB 从一个队列摘下、挂到另一个队列上。
- **索引方式**：按状态分别建立**索引表**（就绪索引表、阻塞索引表等），表中每个表项指向一个 PCB。

*PCB 是整个进程管理中最重要的数据结构。后面的进程控制、调度、同步，本质上都是在读写 PCB。*

## 进程控制的原语

进程控制就是对进程的生命周期进行管理：创建、撤销、阻塞、唤醒、挂起、激活，本质是完成各种状态转换。

进程控制用**原语（Primitive）**实现。原语是一种特殊的程序，具有**原子性**——执行过程"一气呵成"，不可被中断。

原语通过**关中断 / 开中断**指令来实现原子性：进入原语先关中断，使中间过程不会被打断，做完再开中断。关中断、开中断本身是**特权指令**，只能在内核态执行。

### 主要的进程控制原语

- **创建原语**：申请空白 PCB → 为新进程分配资源 → 初始化 PCB → 把进程插入就绪队列。（如系统初始化、用户登录、作业调度、父进程创建子进程时触发）
- **撤销原语**：从队列中找到要撤销进程的 PCB → 回收其占有的资源 → 撤销 PCB。（正常结束、异常结束、外界干预时触发）
- **阻塞原语**：进程主动调用。保存现场 → 把进程状态改为阻塞 → 插入相应阻塞队列 → 转调度程序重新分配 CPU。
- **唤醒原语**：把进程从阻塞队列移出 → 状态改为就绪 → 插入就绪队列。
- **挂起原语 / 激活原语**：在内存与外存之间换出 / 换入进程。

*阻塞和唤醒必须成对使用：谁让它阻塞，就要有人在事件完成后负责唤醒它，否则进程会永远睡下去。*

## 进程同步

### 同步与互斥

并发进程之间存在两种制约关系：

- **同步（直接制约）**：多个进程为完成同一任务需要**相互配合**，规定它们的执行先后次序。例如：进程 A 算出结果，进程 B 才能拿去用。
- **互斥（间接制约）**：多个进程因**争用同一独占资源**，必须**互相排斥**，同一时间只能一个用。例如：两个进程不能同时使用打印机。

### 临界资源与临界区

- **临界资源**：一段时间内只允许一个进程访问的资源（如打印机、共享变量）。
- **临界区**：进程中**访问临界资源**的那段代码。

互斥访问临界区可分为四部分：

```
进入区   ──  检查能否进入，能进入就上锁
临界区   ──  访问临界资源
退出区   ──  解锁，唤醒其他等待进程
剩余区   ──  其余代码
```

### 同步机制应遵循的准则

1. **空闲让进**：临界区空闲时，应允许一个进程进入。
2. **忙则等待**：已有进程在临界区时，其他进程必须等待。
3. **有限等待**：等待进入的进程应在有限时间内能进入，不能无限等（避免**饥饿**）。
4. **让权等待**：进不去时应**释放 CPU**，去阻塞等待，而不是占着 CPU 空转（避免**忙等待**）。

### 信号量机制

信号量（Semaphore）由 Dijkstra 提出，用一个变量 S 配合两个**原子操作** P、V（也叫 wait、signal）来实现同步与互斥。

**整型信号量**：S 是整数，P 操作里用 `while(S<=0)` 空转检查——不满足"让权等待"，会**忙等**。

**记录型信号量**：在整型基础上加一个等待队列 L，进不去就把自己挂到队列里睡眠，满足"让权等待"。这是最常用的形式：

```c
typedef struct {
    int value;            // 资源数目
    struct process *L;    // 等待队列
} semaphore;

// P 操作：申请资源
void P(semaphore *S) {
    S->value--;
    if (S->value < 0) {
        // 资源不够，自我阻塞，挂入等待队列 L
        block(S->L);
    }
}

// V 操作：释放资源
void V(semaphore *S) {
    S->value++;
    if (S->value <= 0) {
        // 还有进程在等，唤醒一个
        wakeup(S->L);
    }
}
```

理解 `value` 的物理含义：

- `value > 0`：表示**还可用的资源数**。
- `value = 0`：资源刚好用完。
- `value < 0`：其绝对值表示**正在等待该资源的进程数**。

**用信号量实现互斥**：设互斥信号量 `mutex = 1`，把临界区夹在 P、V 之间：

```c
semaphore mutex = 1;
// 每个进程：
P(mutex);
    // 临界区
V(mutex);
```

**用信号量实现同步**：设同步信号量初值为 0，让"必须先做的事"做完后 V，让"必须后做的事"先 P 等待。

### 经典同步问题

#### 生产者-消费者问题

生产者往缓冲区放产品，消费者从缓冲区取产品。缓冲区有 n 个，要求：缓冲区满时生产者等待，缓冲区空时消费者等待，且对缓冲区的访问要互斥。

```c
semaphore mutex = 1;   // 互斥访问缓冲区
semaphore empty = n;   // 空闲缓冲区数量
semaphore full  = 0;   // 已用缓冲区数量

// 生产者
while (1) {
    生产一个产品;
    P(empty);          // 申请一个空位
    P(mutex);          // 进缓冲区（互斥）
        放入产品;
    V(mutex);
    V(full);           // 产品数 +1
}

// 消费者
while (1) {
    P(full);           // 申请一个产品
    P(mutex);
        取出产品;
    V(mutex);
    V(empty);          // 空位 +1
    消费这个产品;
}
```

两个 P 操作**顺序不能颠倒**——必须先 P 资源信号量（empty/full）再 P 互斥信号量（mutex）。否则可能出现"拿着锁却没资源，又不放锁"的**死锁**。V 操作的顺序则无所谓。

#### 读者-写者问题

多个读者可以**同时读**，但写者必须**独占**（写的时候不许其他读者或写者进入）。核心是用一个计数器 `count` 记录当前读者数量，配一个 `rmutex` 保护这个计数器：

```c
semaphore rmutex = 1;  // 保护 count
semaphore wmutex = 1;  // 保证写者互斥 / 读写互斥
int count = 0;         // 当前读者数

// 读者
P(rmutex);
    if (count == 0) P(wmutex);  // 第一个读者负责"关门"
    count++;
V(rmutex);
    读文件;
P(rmutex);
    count--;
    if (count == 0) V(wmutex);  // 最后一个读者负责"开门"
V(rmutex);

// 写者
P(wmutex);
    写文件;
V(wmutex);
```

*上面这种写法读者优先，可能让写者**饥饿**。考试常考"为什么需要 rmutex"——因为 count 本身也是被多个读者共享的临界资源。*

#### 哲学家进餐问题

5 个哲学家围圈而坐，相邻两人间有一根筷子，共 5 根。哲学家要拿到**左右两根**筷子才能吃饭。

- **死锁的根源**：如果 5 人同时拿起左手筷子，再去拿右手筷子时全部拿不到，谁也不放，造成**死锁**。
- 常见解法：
  1. 最多允许 4 个哲学家同时去拿筷子；
  2. 仅当左右两根筷子都可用时才允许拿（把拿两根筷子做成一个原子操作）；
  3. 奇数号先拿左、偶数号先拿右，破坏"循环等待"。

## 进程通信

进程通信（IPC）指进程之间交换信息。按数据量分为：

- **低级通信**：只能传递少量信息（如信号量 PV、信号）。
- **高级通信**：以较高效率传送大量数据，用户不必关心底层细节。

高级通信主要有三类：

```
进程通信
├── 共享存储（Shared Memory）
├── 消息传递（Message Passing）
│   ├── 直接通信：消息直接发给目标进程
│   └── 间接通信：消息发到"信箱"，再从信箱取
└── 管道通信（Pipe）
```

- **共享存储**：在内存中开辟一块共享区域，进程通过读写这块区域交换数据。速度快，但需要用同步机制（如信号量）保证互斥访问。
- **消息传递**：以"消息（报文）"为单位，用 `send` / `receive` 两个原语收发。
  - **直接通信**：发送时直接指明接收进程。
  - **间接通信**：消息先送入中间实体**信箱（邮箱）**，接收方再从信箱取走。
- **管道通信**：用一个连接读写进程的共享文件（pipe）传数据，**半双工**、先进先出。写满时写进程等待，读空时读进程等待，需互斥访问。

## 线程

### 为什么引入线程

进程既是**资源分配**的单位，又是**调度**的单位，每次切换进程都要切换地址空间、保存大量现场，开销很大。为了**减小并发执行的开销、提高并发度**，引入了**线程（Thread）**：把"资源分配"和"调度"两件事拆开。

- **进程**：仍是**资源分配**的基本单位。
- **线程**：成为**调度（CPU 分配）**的基本单位。

一个进程内可以有多个线程，它们**共享所属进程的资源**（地址空间、打开的文件等），但每个线程有自己的栈、寄存器、程序计数器和运行状态。

### 进程与线程的比较

| 对比项 | 进程 | 线程 |
|---|---|---|
| 基本单位 | 资源分配 | 处理机调度 |
| 拥有资源 | 拥有独立资源 | 基本不拥有资源，共享进程资源 |
| 切换开销 | 大（要切地址空间） | 小（同进程内不切地址空间） |
| 通信 | 需要 IPC，较复杂 | 直接读写共享数据，简单快 |
| 独立性 | 高 | 低（一个线程崩溃可能拖垮整个进程） |

### 线程的实现方式

- **用户级线程（ULT）**：线程管理全部由用户态的线程库完成，内核感知不到线程。
  - 优点：切换不用陷入内核，开销小，可在不支持线程的 OS 上实现。
  - 缺点：一个线程阻塞（如发起系统调用），**整个进程都被阻塞**；不能利用多核并行。
- **内核级线程（KLT）**：线程的创建、调度由内核完成。
  - 优点：一个线程阻塞不影响同进程其他线程；可在多核上真正并行。
  - 缺点：线程切换要陷入内核，开销较大。
- **组合方式**：把用户级线程映射到内核级线程上（多对一、一对一、多对多），兼顾两者优点。
